## 1. 4단계 스마트 검색 설계

<img src="https://img.shields.io/badge/-비용_최적화-10B981?style=flat-square" alt="비용 최적화" /> <img src="https://img.shields.io/badge/-검색_알고리즘-3178C6?style=flat-square" alt="검색 알고리즘" />

### 문제

FAQ 챗봇에서 사용자 입력에 대한 검색 기능이 필요했다. 모든 질문을 AI에게 던지는 것이 가장 간단하지만, **API 호출당 비용이 발생**하는 구조에서 모든 검색을 AI로 처리하면 비용이 기하급수적으로 증가한다.

### 고려한 방안

| 방안 | 장점 | 단점 |
|:-----|:-----|:-----|
| AI 전수 처리 | 구현 간단, 자연어 이해 | 호출당 비용 발생, 응답 지연 1~3초 |
| 키워드 검색만 | 비용 0, 빠름 | 유사 표현 매칭 불가 |
| **단계별 확대 검색** | 비용 최소화 + 정확도 확보 | 구현 복잡도 증가 |

### 결정: 4단계 점진적 확대 검색

```
사용자 입력
    |
    v
+------------------------------------+
| 1단계: 질문 텍스트 검색 (DB)         |  <-- Supabase ilike + 클라이언트 폴백
|     초성 입력 -> 클라이언트 검색      |
+----------------+-------------------+
                 | 결과 없음
                 v
+------------------------------------+
| 2단계: 서브카테고리 키워드 매칭       |  <-- 동기, 사전 정의 키워드
+----------------+-------------------+
                 | 결과 없음
                 v
+------------------------------------+
| 3단계: 카테고리 키워드 매칭           |  <-- 동기, 더 넓은 범위
+----------------+-------------------+
                 | 결과 없음
                 v
+------------------------------------+
| 4단계: AI Fallback (Claude API)     |  <-- 마지막 수단, 15초 타임아웃
+------------------------------------+
```

**설계 핵심:**
- 1단계는 DB 검색 실패 시 **정적 데이터 클라이언트 검색으로 자동 폴백** -- 서버 장애에도 검색 가능
- 2~3단계는 사전 정의된 키워드 매칭이므로 **네트워크 호출 없이 즉시 응답** (0ms)
- 4단계 AI는 `AbortController`로 **15초 타임아웃**을 강제하여 무한 대기 방지

### 결과

- 대부분의 검색이 1~2단계에서 해결 -- **AI API 호출 비용 대폭 절약**
- 1~3단계 평균 응답 시간: **~100ms** vs AI 단일 처리: **1~3초**
- 서버 장애 시에도 1~3단계 검색 정상 작동 (Graceful Degradation)

## 2. 한글 초성 검색 알고리즘 직접 구현

<img src="https://img.shields.io/badge/-유니코드-764ABC?style=flat-square" alt="유니코드" /> <img src="https://img.shields.io/badge/-외부_의존성_0-10B981?style=flat-square" alt="외부 의존성 0" />

### 문제

한국어 FAQ 챗봇에서 사용자가 `ㅅㄱ`을 입력하면 "**수강**신청"이 검색되어야 한다. 한글 초성 검색은 Supabase의 `ilike` 쿼리로 처리할 수 없다 (초성은 유니코드 조합 문자가 아니기 때문).

### 고려한 방안

| 방안 | 장점 | 단점 |
|:-----|:-----|:-----|
| hangul-js 라이브러리 | 빠른 도입 | 번들 크기 증가, 불필요한 기능 포함 |
| **유니코드 계산 직접 구현** | 의존성 0, 정확한 제어 | 유니코드 구조 이해 필요 |
| 서버 사이드 처리 | 클라이언트 부담 감소 | API 호출 필요, 지연 발생 |

### 결정: 유니코드 수학 공식으로 직접 구현

한글 유니코드(U+AC00 ~ U+D7A3)는 **초성 x 588 + 중성 x 28 + 종성** 규칙으로 배열된다. 이 공식을 활용하면 외부 라이브러리 없이 초성을 추출할 수 있다.

```typescript
/** 한글 초성 배열 (유니코드 순서) */
const CHOSUNG = [
  "ㄱ", "ㄲ", "ㄴ", "ㄷ", "ㄸ", "ㄹ", "ㅁ", "ㅂ", "ㅃ",
  "ㅅ", "ㅆ", "ㅇ", "ㅈ", "ㅉ", "ㅊ", "ㅋ", "ㅌ", "ㅍ", "ㅎ",
];

/** 한글 음절에서 초성 추출: "가" -> "ㄱ", "수" -> "ㅅ" */
const getChosung = (char: string): string => {
  const code = char.charCodeAt(0);
  if (code < 0xAC00 || code > 0xD7A3) return char;
  return CHOSUNG[Math.floor((code - 0xAC00) / 588)];
};

/** 초성 매칭: "ㅅㄱ"이 "수강신청"에 포함되는지 */
const isChosungMatch = (text: string, input: string): boolean => {
  if (!input || !isChosungOnly(input)) return false;
  const textChosung = text.split("").map(getChosung).join("");
  return textChosung.includes(input);
};
```

### 결과

- 전체 알고리즘 **59줄**, 외부 의존성 **0개**
- O(n) 시간 복잡도 (FAQ 데이터 전체 순회)
- 유니코드 표준 기반이므로 **모든 한글 음절 11,172자** 완벽 지원

## 3. Zustand + TanStack Query 이중 상태 설계

<img src="https://img.shields.io/badge/-성능_최적화-EF4444?style=flat-square" alt="성능 최적화" /> <img src="https://img.shields.io/badge/-상태_관리-764ABC?style=flat-square" alt="상태 관리" /> <img src="https://img.shields.io/badge/-서버_상태-FF4154?style=flat-square" alt="서버 상태" />

### 문제

두 가지 문제가 동시에 존재했다:

1. **챗봇**: 단일 스토어에서 타이핑 애니메이션(`isTyping`)이 카테고리 버튼, 검색 결과 등 무관한 컴포넌트까지 리렌더링
2. **관리자**: `useState` + `useEffect`로 Supabase 데이터를 fetch하면, 페이지 전환 시 매번 로딩이 발생하고 mutation 후 수동 리페치 필요

### 고려한 방안

| 방안 | 장점 | 단점 |
|:-----|:-----|:-----|
| React Context API | React 내장 | Provider 중첩, context 변경 시 하위 전체 리렌더 |
| Redux Toolkit | 성숙한 생태계 | 보일러플레이트 과다, FAQ 챗봇 규모에 과도 |
| Zustand 단일 스토어 | 간결 | 상태 간 결합으로 불필요한 리렌더링 |
| **Zustand + TanStack Query** | 클라이언트/서버 상태 분리, 각각 최적화 | 두 시스템 학습 필요 |

### 결정: Zustand 5-Store (클라이언트) + TanStack Query (서버)

**챗봇 -- Zustand 5개 독립 스토어:**

```
+-----------------------------------------------------+
|                  Zustand Stores                      |
+----------+----------+----------+----------+---------+
| message  |navigation| search   | faq-data | company |
| messages | current  | search   | faqData  | config  |
| isTyping | Category | Results  | companyId| theme   |
+----------+----------+----------+----------+---------+
```

```typescript
// 셀렉터 훅: 필요한 필드만 구독
const messages = useMessages();       // messages만 구독
const isTyping = useIsTyping();       // isTyping만 구독
```

**관리자 -- TanStack Query v5 서버 상태 캐싱:**

```typescript
// Query Key Factory -- 계층적 키로 세밀한 캐시 무효화
const adminKeys = {
  all: ["admin"],
  categories: (companyId) => [...adminKeys.all, "categories", companyId],
  suggestions: (companyId) => [...adminKeys.all, "suggestions", companyId],
};

// staleTime: Infinity -- mutation이 캐시 갱신을 전담
const adminQueryClient = new QueryClient({
  defaultOptions: { queries: { staleTime: Infinity, gcTime: 5 * 60_000 } },
});

// mutation 성공 시 관련 캐시만 정확히 무효화
useMutation({
  mutationFn: updateCompanySettings,
  onSuccess: () =>
    queryClient.invalidateQueries({ queryKey: adminKeys.company(companyId) }),
});
```

**역할 분리 원칙:**
- **Zustand** = 클라이언트 상태 (메시지, 네비게이션, 검색, UI 상태)
- **TanStack Query** = 서버 상태 (FAQ, 설정, 분석, 회사 관리, 추천 검색어 -- 5개 도메인)
- `staleTime: Infinity`로 Admin 데이터는 mutation으로만 갱신 -- 불필요한 리페치 제거

### 결과

- `isTyping` 변경 시 **MessageBubble만 리렌더링**, 카테고리 버튼/검색 UI 영향 없음
- Admin 페이지 전환 시 **캐시 히트로 즉시 렌더링** (로딩 스피너 제거)
- FAQ, 설정, 분석, 회사 관리, 추천 검색어 5개 도메인에 걸쳐 일관된 캐싱 전략

## 4. Tree Diff/Save 패턴 (FAQ 편집 최적화)

<img src="https://img.shields.io/badge/-DB_최적화-F59E0B?style=flat-square" alt="DB 최적화" /> <img src="https://img.shields.io/badge/-알고리즘-3178C6?style=flat-square" alt="알고리즘" />

### 문제

관리자 페이지에서 FAQ를 편집할 때, 카테고리 > 서브카테고리 > 질문으로 이루어진 **3단 트리 구조**를 관리해야 한다. 1000개 질문 중 1개만 수정해도 전체를 저장하면 불필요한 DB 쿼리가 999개 발생한다.

### 고려한 방안

| 방안 | 장점 | 단점 |
|:-----|:-----|:-----|
| 전체 저장 (DELETE + INSERT ALL) | 구현 간단 | 1000개 질문 -> 1000개 쿼리 |
| 즉시 저장 (수정 즉시 API 호출) | 실시간 반영 | 편집 중 네트워크 에러 시 불일치 |
| **Diff 저장 (원본 vs 현재 비교)** | 변경분만 저장 | 구현 복잡, Diff 알고리즘 필요 |

### 결정: Git 스타일 Tree Diff 알고리즘

```
원본 트리 (DB 로드 시점)              현재 트리 (편집 후)
+--------------+                     +--------------+
| 카테고리 A    |                     | 카테고리 A    |  (label 변경)
|  +- 서브 1   |                     |  +- 서브 1   |
|  |  +- Q1    |                     |  |  +- Q1    |
|  |  +- Q2    |                     |  |  +- Q2    |  (answer 수정)
|  +- 서브 2   |                     |  +- 서브 3   |  <-- 새로 추가 (temp_id)
| 카테고리 B    |                     +--------------+  <-- B 삭제
+--------------+

                     | computeTreeDiff()
                     v
               +----------------------------+
               | deletedCategories: ["B"]
               | createdSubCategories: {"temp_xxx": 서브3}
               | updatedCategories: {"A": {label}}
               | updatedQuestions: {"Q2": {answer}}
               +----------------------------+
```

**구현 핵심:**
1. **평탄화(flatten)**: 3단 트리를 `Map<id, data>` 3개로 변환
2. **Diff 계산**: 원본 Map과 현재 Map을 비교하여 삭제/생성/수정 분류
3. **임시 ID**: 새 아이템은 `temp_` 접두사 UUID -- DB 저장 시 실제 ID로 교체
4. **미로드 보호**: `loaded: false`인 카테고리의 자식은 Diff에서 제외

### 결과

- 1000개 중 1개 수정 시 -- **DB 쿼리 1개** (1000개 대신)
- 트리 비교 함수 단위 테스트 작성 완료 (`tree-diff.test.ts`)
- DnD 드래그 정렬 시 `sort_order` 변경분만 정확히 추적

## 5. RAG + 멀티테넌트 격리

<img src="https://img.shields.io/badge/-보안-EF4444?style=flat-square" alt="보안" /> <img src="https://img.shields.io/badge/-AI_아키텍처-D97757?style=flat-square" alt="AI 아키텍처" />

### 문제

여러 회사가 하나의 플랫폼을 사용하므로, A회사의 FAQ로 학습된 AI가 B회사 사용자에게 A회사 정보를 노출하면 안 된다. 동시에 각 회사별로 AI 모델을 Fine-tuning하는 것은 비용과 관리 측면에서 비현실적이다.

### 결정: 회사별 FAQ 동적 주입 RAG

```
사용자 질문 --> Edge Function (chat-ai)
                       |
                       v
            +---------------------------+
            | 1. 회사 ID로 설정 조회      |  <-- ai_enabled, ai_model, ai_system_prompt
            | 2. 해당 회사 FAQ 조회       |  <-- company_id 기준 격리
            | 3. 시스템 프롬프트 조합      |  <-- 커스텀 프롬프트 + FAQ 컨텍스트
            | 4. Claude API 호출         |  <-- 회사별 모델/온도/토큰 설정 적용
            +---------------------------+
```

**보안 계층:**

| 계층 | 구현 |
|:-----|:-----|
| **세션 격리** | 챗봇은 `supabase-public.ts` (persistSession: false) -- 관리자 세션과 완전 분리 |
| **Rate Limiting** | IP당 분당 20회 제한 -- API 비용 남용 방지 |
| **타임아웃** | 클라이언트 `AbortController` 15초 + Edge Function 자체 타임아웃 |
| **입력 제한** | 메시지 500자 제한 -- 프롬프트 인젝션 공격 표면 축소 |
| **이중 검증** | `ai_enabled` + `ai_subscription_status` 모두 확인 |
| **데이터 격리** | FAQ 조회 시 `company_id` WHERE 조건 -- 타사 데이터 접근 원천 차단 |

### 결과

- 회사 추가 시 **코드 수정 0** -- DB에 회사 레코드 추가만으로 독립 챗봇 운영
- FAQ 수정 즉시 AI 응답에 반영 (재학습/배포 불필요)
- 회사 간 데이터 완전 격리 확인 (다중 보안 계층)

## 6. Supabase 폴백 + On-demand 지연 로딩

<img src="https://img.shields.io/badge/-장애_복원력-10B981?style=flat-square" alt="장애 복원력" /> <img src="https://img.shields.io/badge/-성능_최적화-EF4444?style=flat-square" alt="성능 최적화" />

### 문제

챗봇 초기 로딩 시 모든 FAQ 데이터를 한 번에 가져오면 **초기 로딩 시간이 길어지고**, Supabase 서버 장애 시 **챗봇 전체가 작동 불능**이 된다.

### 결정: 2단계 On-demand 로딩 + 정적 데이터 폴백

```
초기 로드 (카테고리 목록만)
    |
    v
+---------------------------+
| fetchCategoryList()       |  --> 카테고리 이름만 fetch (질문 없음)
| + fetchCompanyId()        |  --> 로깅/AI용 companyId 확보
+-----------+---------------+
            |
            v  사용자가 카테고리 클릭
+---------------------------+
| 1단계: loadCategoryData() |  --> 서브카테고리 fetch (질문 제외)
+-----------+---------------+
            |
            v  사용자가 서브카테고리 클릭
+---------------------------+
| 2단계: loadSubCatData()   |  --> 해당 서브카테고리의 질문만 fetch
+---------------------------+

폴백 체인:
  Supabase fetch 성공 --> DB 데이터 사용
  Supabase fetch 실패 --> 정적 데이터(staticFaqData) 자동 전환
```

### 결과

- 초기 로딩: 카테고리 이름만 fetch -- **데이터 전송량 최소화**
- Supabase 장애 시에도 정적 데이터 폴백으로 **챗봇 기본 기능 유지**
- `Set` 기반 중복 체크로 같은 카테고리 **재요청 방지**

## 7. Vite 번들 최적화

<img src="https://img.shields.io/badge/-빌드-646CFF?style=flat-square" alt="빌드" /> <img src="https://img.shields.io/badge/-성능_최적화-EF4444?style=flat-square" alt="성능 최적화" />

### 문제

챗봇과 관리자 페이지가 하나의 SPA에 포함되어 있다. 최적화 없이 빌드하면 **챗봇 사용자가 TipTap, recharts, dnd-kit 등 관리자 전용 라이브러리까지 로드**하게 된다.

### 결정: React.lazy 코드 분할 + Vite manualChunks

```javascript
// vite.config.ts
manualChunks: {
  "vendor-react":    ["react", "react-dom", "react-router-dom"],
  "vendor-zustand":  ["zustand"],
  "vendor-tiptap":   ["@tiptap/react", "@tiptap/starter-kit"],  // 관리자 전용
  "vendor-recharts": ["recharts"],                                // 관리자 전용
  "vendor-dnd":      ["@dnd-kit/core", "@dnd-kit/sortable"],     // 관리자 전용
  "vendor-framer":   ["framer-motion"],
}
```

```
챗봇 사용자가 로드하는 청크:          관리자가 추가로 로드하는 청크 (lazy):
+-- vendor-react     (공유)          +-- vendor-tiptap    (리치 텍스트)
+-- vendor-zustand   (공유)          +-- vendor-recharts  (분석 차트)
+-- vendor-framer    (공유)          +-- vendor-dnd       (FAQ 정렬)
+-- chatbot chunk    (챗봇)          +-- admin chunk      (관리자 코드)
```

### 결과

- 챗봇 초기 번들에서 Admin 코드 완전 제거 -- **~58% 번들 크기 절감**
- 6개 벤더 청크로 분리 -- 라이브러리 업데이트 시 **해당 청크만 캐시 무효화**

## 8. 서버 인프라 직접 구축

<img src="https://img.shields.io/badge/-NGINX-009639?style=flat-square&logo=nginx&logoColor=white" alt="NGINX" /> <img src="https://img.shields.io/badge/-인프라-F38020?style=flat-square" alt="인프라" />

### 문제

Vercel 같은 매니지드 플랫폼은 편리하지만, 서버 인프라에 대한 **실무 경험이 쌓이지 않는다**. 또한 SaaS 특성상 서브도메인 와일드카드 처리, SSL 종단, CDN 캐싱 등을 직접 제어할 필요가 있었다.

### 고려한 방안

| 방안 | 장점 | 단점 |
|:-----|:-----|:-----|
| Vercel | 제로 설정, 자동 배포 | 인프라 학습 없음, 서브도메인 제한 |
| AWS EC2 | 완전한 제어 | 비용 높음, 과도한 복잡도 |
| **가비아 + NGINX** | 저비용, 실무 인프라 경험 | 수동 설정 필요 |

### 결정: 가비아 호스팅 + NGINX + Cloudflare

```
배포 파이프라인:
  npm run build --> dist/ 생성
    --> SCP로 서버 업로드 (root@서버:/var/www/chatbot/)
      --> NGINX가 정적 파일 서빙
        --> Cloudflare가 SSL/CDN/DDoS 처리

NGINX 설정:
  +-- try_files $uri $uri/ /index.html  (SPA 클라이언트 라우팅)
  +-- gzip 압축 (JS/CSS/HTML)
  +-- 정적 파일 캐싱 헤더
  +-- 서브도메인 와일드카드 (*.goofitalk.co.kr)
```

**직접 수행한 작업:**
- SSH 접속, 파일 권한 설정 (`chmod 755`, `chown nginx:nginx`)
- SELinux 컨텍스트 설정 (`chcon`)
- Cloudflare DNS -- 프록시 -- NGINX 오리진 서버 구조 설계
- `deploy.sh` 스크립트로 빌드 -- 업로드 -- 권한 설정을 원커맨드로 자동화

### 결과

- SSH, 파일 권한, SELinux, NGINX 설정 등 **서버 인프라 실무 경험 확보**
- Cloudflare 프록시를 통한 **SSL 종단 + CDN 캐싱 + DDoS 방어** 구성
- 매니지드 플랫폼 없이 **안정적인 프로덕션 운영 환경** 구축

## 9. AI 기반 개발 프로세스

<img src="https://img.shields.io/badge/-Claude_Code-D97757?style=flat-square&logo=anthropic&logoColor=white" alt="Claude Code" /> <img src="https://img.shields.io/badge/-Shrimp_Task_MCP-191919?style=flat-square" alt="Shrimp Task MCP" />

> Claude Code는 **도구**로 활용했다. 핵심은 AI가 프로젝트의 컨벤션과 아키텍처를 정확히 준수하도록 **시스템화한 것**이다.

### 프로젝트 규칙 시스템 (.claude/rules/)

AI 코딩 도구의 가장 큰 문제는 프로젝트 맥락 없이 일반적인 코드를 생성하는 것이다. 이를 해결하기 위해 **600줄 이상의 프로젝트 규칙 파일**을 설계했다.

규칙 파일에 포함된 항목:
- **디렉토리별 코드 배치 규칙** -- 새 기능이 어디에 위치해야 하는지 Decision Tree로 정의
- **Zustand 사용 패턴** -- 셀렉터 훅 강제, 전체 구독 금지, get 접근자 사용 조건
- **메시지 추가 시퀀스** -- `clearAllDisplayTypes()` -> `addUserMessage()` -> `showTypingThen()` 순서 강제
- **멀티파일 동시 수정 규칙** -- 카테고리 추가 시 4개 파일 동시 수정 등 연쇄 변경 명시
- **보안 필수 규칙** -- HTML 살균, Supabase 클라이언트 구분, 분석 로그 fire-and-forget

### 태스크 관리 워크플로우

Shrimp MCP(태스크 매니저)를 연동하여 체계적인 개발 흐름을 구성했다:

```
plan_task --> analyze_task --> split_tasks --> execute_task --> verify_task
```

### 핵심 메시지

Claude Code는 생산성을 높여주는 도구이지, 개발 실력을 대체하는 것이 아니다. **위 8가지 기술적 결정은 모두 직접 설계한 것**이고, Claude Code는 그 설계를 코드로 구현하는 과정에서 활용했다. 도구를 효과적으로 사용하려면 **무엇을 만들어야 하는지 정확히 알고 있어야** 하며, 그 "무엇"을 정의하는 규칙 시스템 자체가 개발자의 역량을 보여준다.
